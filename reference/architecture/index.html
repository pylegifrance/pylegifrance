    Architecture de PyLegifrance

    Ce document explique l'architecture du projet PyLegifrance, en se concentrant particulièrement sur la structure des modules fonds/juri.py et models/juri/.

    Structure du projet

    L'architecture de PyLegifrance suit une approche de conception en couches qui sépare les préoccupations et facilite la maintenance et l'évolution du code.

    Rationale de la structure fonds/juri.py et models/juri/

    La structure du code avec fonds/juri.py et models/juri/ illustre une séparation claire entre les façades d'API de haut niveau et les modèles de données sous-jacents. Cette séparation suit les principes de conception orientée domaine (DDD) et de séparation des préoccupations.

    Explication des composants

    Couche Façade (fonds/juri.py)

    Le répertoire fonds contient des façades de domaine qui organisent l'API publique de Legifrance par fonds (LEGI, JORF, KALI, JURI, CNIL). Cette organisation présente plusieurs avantages :

    1. Cohérence fonctionnelle : Les routes REST de l'API Legifrance sont regroupées dans des contrôleurs techniques (Consult, Search, List, Chrono, etc.) sans cohérence fonctionnelle pour l'utilisateur. Les façades dans fonds réorganisent ces fonctionnalités selon une logique métier.

    2. Simplicité d'utilisation : Les praticiens du droit raisonnent en termes de fonds. Les rassembler sous une même arborescence simplifie la découverte et l'autocomplétion.

    3. Stabilité de l'interface : Les points d'extrémité exposés par ce module peuvent être renommés ou dépréciés sans rupture de contrat. L'interface publique (façades) reste stable et redirige automatiquement vers les implémentations de remplacement.

    4. Orchestration transparente : Certaines opérations juridiques nécessitent plusieurs appels réseau. Une seule méthode haut niveau (par exemple Article.at(date)) les orchestre de manière transparente.

    Dans juri.py, nous trouvons :

    • JuriAPI : Une classe qui fournit des méthodes pour interagir avec l'API Legifrance pour les données de jurisprudence (fetch, search, etc.).
    • JuriDecision : Une classe qui représente une décision de jurisprudence avec des méthodes pour accéder à ses propriétés et versions.

    Couche Modèles (models/juri/)

    Le répertoire models/juri/ contient les modèles de données pour la jurisprudence, qui sont utilisés par la façade JuriAPI pour interagir avec l'API Legifrance. Cette séparation permet :

    1. Réutilisation : Les modèles peuvent être utilisés indépendamment de la façade.
    2. Testabilité : Les modèles peuvent être testés séparément de la logique d'API.
    3. Évolutivité : Les modèles peuvent évoluer indépendamment de la façade.

    Dans models/juri/, nous trouvons :

    • base.py : Définit JuriBaseModel, une classe de base pour tous les modèles JURI avec une configuration cohérente.
    • models.py : Définit Decision, un modèle de domaine pour les décisions judiciaires.
    • api_wrappers.py : Définit des classes qui encapsulent les requêtes et réponses de l'API.
    • search.py : Définit des modèles pour la recherche de jurisprudence.
    • constants.py : Définit des constantes et énumérations utilisées dans les modèles.
    • filters.py : Définit des filtres pour la recherche.

    Flux de données

    1. L'utilisateur interagit avec les classes de façade (JuriAPI, JuriDecision).
    2. La façade utilise les modèles de données pour créer des requêtes API.
    3. Les requêtes sont envoyées à l'API Legifrance via LegifranceClient.
    4. Les réponses de l'API sont converties en modèles de données.
    5. Les modèles de données sont utilisés pour créer des objets de domaine (JuriDecision).
    6. Les objets de domaine sont retournés à l'utilisateur.

    Cette architecture en couches permet une séparation claire des préoccupations, une meilleure testabilité et une évolution plus facile du code.
